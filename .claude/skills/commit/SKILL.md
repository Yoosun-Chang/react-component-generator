  사용자가 "커밋", "커밋해줘", "변경사항 저장해줘", "저장해줘", "commit", "변경사항 커밋" 등의 표현을 쓸 때 반드시 이 스킬을 사용합니다.
  git 변경사항 분석, 논리적 단위 분류, 한국어 커밋 메시지 작성, 사용자 승인 후 커밋까지 전 과정을 처리합니다.
---

## 목적

변경사항을 분석해서 의미 있는 단위로 나누고, 팀이 이해하기 쉬운 한국어 커밋 메시지를 작성한다.
단순히 파일을 나열하는 게 아니라, "왜 이 변경이 필요했는가"를 메시지에 담는 것이 핵심이다.

---

## 절차

### 1단계: 변경사항 파악

다음 명령어를 병렬로 실행한다:
- `git status` — 변경된 파일 목록 (staged/unstaged/untracked 구분)
- `git diff` — unstaged 변경 내용
- `git diff --staged` — 이미 staged된 변경 내용

변경사항이 없으면 사용자에게 "커밋할 변경사항이 없습니다"라고 알린다.

### 2단계: 논리적 단위로 분류

파일들을 변경 목적에 따라 묶는다. 분류 기준:

- **같은 기능을 위한 변경** → 하나의 커밋
- **서로 독립적인 수정** → 별도 커밋
- **설정 파일 변경** → 기능 변경과 분리 고려

하나의 커밋으로 묶어도 될지 여러 커밋으로 나눌지는 변경 규모와 성격으로 판단한다.
변경 파일이 3개 이하이거나 모두 같은 목적이면 보통 하나로 묶는다.

### 3단계: 커밋 메시지 작성

각 커밋 단위에 대해 아래 형식으로 메시지를 작성한다:

```
<타입>: <한국어 요약>
```

**타입 선택 기준:**
| 타입 | 사용 상황 |
|------|-----------|
| `feat` | 새 기능 추가, 새 컴포넌트, 새 API 엔드포인트 |
| `fix` | 버그 수정, 오타 수정, 잘못된 동작 교정 |
| `refactor` | 동작은 그대로, 코드 구조/가독성 개선 |
| `style` | 포매팅, 공백, 세미콜론 등 (로직 무관) |
| `chore` | 빌드 설정, 의존성, 환경 설정 변경 |
| `docs` | 문서, 주석, README 수정 |
| `test` | 테스트 추가/수정 |

**좋은 메시지 예시:**
- `feat: 컴포넌트 카드에 재생성 버튼 추가`
- `fix: API 키 미입력 시 오류 메시지 미표시 버그 수정`
- `refactor: useComponentGenerator 훅 상태 관리 단순화`
- `chore: Vite 설정에 프록시 경로 추가`

**피해야 할 메시지:**
- `fix: 수정` (너무 모호)
- `feat: 여러 가지 변경` (범위가 불명확)
- `update files` (영어 + 정보 없음)

### 4단계: 사용자에게 제안 제시

변경 요약과 제안된 커밋을 보기 좋게 보여준다:

```
📋 변경사항 분석 결과

[커밋 1/2]
대상 파일:
  - src/components/ComponentCard.tsx
  - src/hooks/useComponentGenerator.ts
메시지: feat: 컴포넌트 재생성 기능 구현

[커밋 2/2]
대상 파일:
  - vite.config.ts
메시지: chore: 개발 서버 프록시 설정 추가

이대로 커밋할까요? (수정이 필요하면 말씀해 주세요)
```

### 5단계: 승인 후 커밋 실행

사용자가 승인하면:

1. 각 커밋 단위별로 해당 파일만 `git add` → `git commit` 순서로 실행
2. 커밋이 여러 개라면 순서대로 처리
3. 완료 후 `git log --oneline -5`로 결과 확인

사용자가 메시지 수정을 요청하면 수정 후 다시 확인을 받는다.

---

## 주의사항

- `.env`, 비밀키, 민감 정보가 포함된 파일은 커밋 전 반드시 경고한다
- `node_modules/`, 빌드 아웃풋 등 `.gitignore`에 있어야 할 파일이 포함되면 알린다
- `--no-verify`는 사용자가 명시적으로 요청할 때만 사용한다
- force push는 사용자 명시 요청 없이는 절대 하지 않는다
